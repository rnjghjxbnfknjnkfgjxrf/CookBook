# WorkingSchedule

## Package
```
Данный проект является серверным веб-приложением, разрабатываемый мной по мере написания лабораторных работ
для курса веб разработки в моем университете.
#   Архитектура проекта
В основе архитектуры лежит так называемый слой запросов:
при получении GET/POST запросов сервер отправляет их в маршрутизатор (router), где каждый путь связан методом GET или POST
с нужным обработчиком (handler), внутри которого формируется нужный ответ для пользователя. Если для ответа требуется информация, находящаяся в 
"базе данных" сервера, то обработчик обращается к  нужным запросам (query), в которых реализована логика обработки данных (работа с репозиториями).
#   Работа с "базой данных"
Вся нужная информация хранится в двух json документах - storage (основные данные) и settings (там хранится соль, используемая для jwt токена и 
хранения паролей). При запуске приложения считывается основной json документ, из которого данные парсятся и отправляются в репозитории, где они хранятся
в виде Map<UUID, сущность>. Запросы берут нужные для обработчиков данные из репозиториев. По окончании работы приложения идет полная перезапись json
документа с использованием данных из репозиториев.
#   Немного о функционале
Сам по себе сайт не обладает обширным функционалом - вcего  присутствует 3 основные сущности - рецепт, ингредиент и автор, сайт же предоставляет
возможность взаимодействия с ними (создание, редактирование и т.д.).
Для аутентификации и авторизации исползуется jwt токен, который хранится в виде cookie файла. На основе этого реализована система ролей:
-гость может только просматривать списки сущностей;
-авторизованный пользователь может добавлять новые рецепты, редактировать существующие собственные и отправлять на модерацию новые ингредиенты;
-администратор может редактировать любые рецепты, ингредиенты, а также одобрять последние.
```
